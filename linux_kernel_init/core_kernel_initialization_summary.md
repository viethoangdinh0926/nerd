`start_kernel()` is the kernel’s architecture-neutral “main” function in `init/main.c`. In current upstream Linux, it begins by setting a stack sentinel on the boot task, identifying the boot CPU, doing very early debug/build-id setup, initializing cgroups early, and then explicitly disabling local interrupts while marking that the system is still in the “early boot, IRQs must stay off” phase. ([codebrowser.dev][1])

A good way to read it is as a sequence of bootstrapping layers:

First, it establishes the boot CPU’s basic identity and machine-specific foundation. `boot_cpu_init()` marks the bootstrap processor as present/online for the early CPU masks, `page_address_init()` prepares early page-address helpers, `pr_notice("%s", linux_banner)` prints the banner, and `setup_arch(&command_line)` hands control to architecture-specific code to finalize low-level setup and return the kernel command line pointer. Right after that, it initializes jump labels and static calls, then runs `early_security_init()`, because later security plumbing depends on those patching mechanisms being available. ([codebrowser.dev][1])

Then it turns raw bootloader input into kernel-owned boot state. `setup_boot_config()` processes bootconfig data if present, `setup_command_line()` preserves both the untouched and mutable command line copies, `setup_nr_cpu_ids()` finalizes how many CPUs the kernel thinks it may manage, `setup_per_cpu_areas()` allocates the per-CPU memory regions, `smp_prepare_boot_cpu()` runs arch-specific bootstrap-CPU hooks, `early_numa_node_init()` seeds early NUMA node mappings, and `boot_cpu_hotplug_init()` prepares the boot CPU’s hotplug state. In practice, this is the point where the kernel stops merely inheriting boot parameters and starts owning them. ([codebrowser.dev][1])

Next comes parameter parsing. The kernel logs the command line, parses “early” parameters first with `parse_early_param()`, then parses the full static command line with `parse_args()`. Anything after `--` is treated as init-process arguments, and unknown options that are not kernel parameters are preserved and later passed toward userspace init. This split is important: early parameters can influence subsystems that must be configured before the general allocator, IRQ setup, or tracing are fully alive. ([codebrowser.dev][1])

After command-line handling, `start_kernel()` seeds randomness and brings up the earliest infrastructure that still needs boot-memory style allocation. `random_init_early(command_line)` performs architectural and non-timekeeping RNG setup before the normal allocators are ready. Then `setup_log_buf(0)` allocates the printk ring buffer, `vfs_caches_init_early()` builds the earliest VFS caches, `sort_main_extable()` sorts the exception table used for fault fixups, `trap_init()` installs the architecture trap/exception setup, `mm_core_init()` brings up core memory-management structures, `maple_tree_init()` initializes the maple tree infrastructure now used widely in memory management, `poking_init()` prepares early text patching machinery, and `ftrace_init()` plus `early_trace_init()` establish tracing foundations. These calls are all still part of “build enough kernel substrate so later subsystems have something to stand on.” ([codebrowser.dev][1])

Once those foundations exist, the kernel initializes scheduling before enabling interrupts. The code comment in `start_kernel()` explicitly says the scheduler must be up before timer interrupts begin arriving. `sched_init()` gives the kernel a functioning scheduler, even though full SMP topology is deferred until later. The kernel then defensively checks whether interrupts were enabled too early and disables them again if necessary. `radix_tree_init()` follows, then `housekeeping_init()` sets up which CPUs are reserved for core kernel “housekeeping” duties, and `workqueue_init_early()` allows early workqueue creation even though actual worker-thread execution comes later. ([codebrowser.dev][1])

The next block brings up core deferred-execution and concurrency primitives. `rcu_init()` initializes classic RCU, `kvfree_rcu_init()` brings up the kvfree-RCU path, and `trace_init()` enables trace events proper. If `initcall_debug` was requested, the kernel enables extra initcall tracing. `context_tracking_init()` initializes context-tracking state, then the interrupt and time stack comes online in a careful order: `early_irq_init()`, `init_IRQ()`, `tick_init()`, `rcu_init_nohz()`, `timers_init()`, `srcu_init()`, `hrtimers_init()`, `softirq_init()`, `timekeeping_init()`, and finally `time_init()`. This order matters because once interrupts and timers are alive, the kernel needs scheduling, RCU, softirqs, and clock infrastructure ready to service them correctly. ([codebrowser.dev][1])

Only after the timekeeping core is initialized does `start_kernel()` do the “full” randomness bring-up with `random_init()`. Then it initializes consumers of high-quality randomness such as `kfence_init()` and refreshes the boot stack canary with `boot_init_stack_canary()`. That distinction between `random_init_early()` and `random_init()` is intentional: some randomness is needed very early, but subsystems that rely on a fully initialized RNG must wait until the timing base is ready. ([codebrowser.dev][1])

The kernel then prepares several CPU-wide and profiling facilities before finally allowing interrupts. `perf_event_init()`, `profile_init()`, and `call_function_init()` come up while interrupts are still expected to be off. Then `early_boot_irqs_disabled` is cleared and `local_irq_enable()` finally enables local interrupts for real. This is a major transition point: from here on, the kernel is no longer in the special “boot processor with hard IRQs off” regime that the early-boot comments describe. ([codebrowser.dev][1])

With interrupts enabled, `start_kernel()` moves into the “late core init” phase. `kmem_cache_init_late()` completes slab-cache setup, `console_init()` turns on the console subsystem earlier than normal device bring-up for visibility, and if command-line parsing overflowed the init/env arrays, the kernel panics immediately. It then initializes lock dependency checking with `lockdep_init()` and runs `locking_selftest()`, which specifically needs IRQs enabled because it tests lock-inversion cases involving hard and soft IRQ contexts. ([codebrowser.dev][1])

After that, it validates initrd placement and starts bringing up core memory/process/filesystem/accounting subsystems in earnest. It checks that initrd was not overwritten, then runs `setup_per_cpu_pageset()`, `numa_policy_init()`, `acpi_early_init()`, any architecture-supplied `late_time_init()`, `sched_clock_init()`, and `calibrate_delay()`. Then it moves through process and namespace foundations: `arch_cpu_finalize_init()`, `pid_idr_init()`, `anon_vma_init()`, `thread_stack_cache_init()`, `cred_init()`, `fork_init()`, `proc_caches_init()`, `uts_ns_init()`, `time_ns_init()`, `key_init()`, `security_init()`, `dbg_late_init()`, `net_ns_init()`, `vfs_caches_init()`, `pagecache_init()`, `signals_init()`, `seq_file_init()`, `proc_root_init()`, `nsfs_init()`, `pidfs_init()`, `cpuset_init()`, `mem_cgroup_init()`, `cgroup_init()`, `taskstats_init_early()`, and `delayacct_init()`. By the end of this block, the machine still has not started userspace, but memory management, process creation, namespaces, procfs roots, cgroups, and security are largely in place. ([codebrowser.dev][1])

Right before leaving `start_kernel()`, the kernel initializes the ACPI subsystem proper with `acpi_subsystem_init()`, gives the architecture a post-ACPI hook with `arch_post_acpi_subsys_init()`, starts the Kernel Concurrency Sanitizer if configured via `kcsan_init()`, and then calls `rest_init()`. `rest_init()` is the handoff out of the giant single-threaded initialization routine: it creates the first kernel threads, eventually spawns the kernel init thread and `kthreadd`, transitions the system state toward scheduling, and starts the normal task-scheduling world. ([codebrowser.dev][1])

The best mental model is that `start_kernel()` is not “one initializer.” It is a ladder:

1. establish CPU/arch ownership,
2. capture boot parameters,
3. seed the minimal runtime,
4. bring up memory, traps, scheduler, RCU, IRQs, and time,
5. enable interrupts,
6. finish process/filesystem/security/cgroup foundations,
7. hand off to `rest_init()` so the scheduler and kernel threads can take over. ([codebrowser.dev][1])

Two caveats matter when reading it. First, the exact call list varies by kernel version and config; the walkthrough above matches current upstream structure in `init/main.c`, not every historic kernel. Second, several of these calls are only thin orchestration points: the real work fans out into many other source files, especially `setup_arch()`, `trap_init()`, `mm_core_init()`, `sched_init()`, `init_IRQ()`, `time_init()`, and `security_init()`. ([codebrowser.dev][1])
